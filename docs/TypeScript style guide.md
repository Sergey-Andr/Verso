В этом разделе описаны правила работы с типами в проекте.  
⚠️ Вопрос расположения файлов см. в разделе [[Architecture/00_Overview|Architecture]].

---

## Основная концепция

- Применяем **принцип близости**:  
  - типы, используемые **только в одном месте**, остаются в том же файле рядом с кодом;  
  - **переиспользуемые типы** выносим в общий файл/папку на уровне, доступном всем использующим их компонентам.

- Избегаем дублирования: всегда проверяй, нет ли уже существующего типа, который подходит под задачу.

---

## Типизация компонентов

- Если у компонента **≤ 3 пропсов** → используем **inline-типизацию**:  

  ```tsx
  function Button({ label, disabled }: { label: string; disabled?: boolean }) { 
	  ...
  }
  ```

- Если у компонента **> 3 пропсов** → создаём отдельный тип `ComponentNameProps`:
```tsx
type ButtonProps = {
  label: string;
  disabled?: boolean;
  onClick: () => void;
  isOpen: boolean;
};

function Button(props: ButtonProps) {
  ...
}
```

## Типизация стейтов и переменных

- Для **простых значений** используем примитивы напрямую:
    
  ```jsx
      const [isOpen, setIsOpen] = useState(false); // boolean 
      const [count, setCount] = useState(0);       // number
    ```
    
- Если состояние хранит **объект или структуру с несколькими полями** → обязательно создаём отдельный тип:
    
   ```tsx
    type UserState = {
	    id: string;   
	    name: string;   
	    isActive: boolean; 
    };  
    
    const [user, setUser] = useState<UserState | null>(null);
    ```
    
- Для массивов и коллекций всегда явно указываем дженерик:
    
   ```tsx
    const [users, setUsers] = useState<UserState[]>([]);
    ```

## Переиспользование типов

- Переиспользуем типы максимально — особенно для API-ответов, моделей и данных, которые идут сквозным потоком по приложению.
    
-  Если `boolean`, `string`, `number` и т. п. используются **только внутри одного компонента** → их **не выносим**.
    
	 - Если простой тип повторяется в **родительском и дочернем компоненте** (например, id: `string`) → всё равно **не нужно выносить в отдельный тип**.
		   Такие базовые типы считаем _самоочевидными_.
	    
	 - Вынос в отдельный тип (`type Id = string`) делаем только если:	значение имеет **особый бизнес-смысл** (например, `UserId` или `OrderId`);
		   Или если этот тип используется в **нескольких разных местах проекта** и может поменяться в будущем.
	    
- Для дочерних компонентов стараемся прокидывать существующие типы пропсов или их подтипы через `Pick` / `Omit` / `Partial`, а не дублировать:
    
  ```tsx
    type CardProps = {   
	    id: string;   
	    title: string;   
	    description?: string; 
    };  
    
    type CardHeaderProps = Pick<CardProps, "title">;
    ```

## Дополнительные правила

- Для **API**:
    
    - выносим типы запросов и ответов в `/types.ts`;
        
    - всегда разделяем входные и выходные данные (`FetchCitiesRequest`, `FetchCitiesResponse`).
        
- Для **enum и union типов**:
    
    - используем `union` строк/чисел, если набор небольшой и жёстко фиксированный;
        
    - используем `enum` только если нужно автодополнение или расширяемость.
