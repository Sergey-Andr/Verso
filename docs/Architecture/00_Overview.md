# Краткий обзор

Проект написан на `Nextj.js`, проект начал писаться до того как браузеры научились индексировать клиентские компоненты,
и
по этому в проекте могут встречаться как `server` так и `client side` компоненты в неочевидных местах.

> ✍️ **Заметка**
> Для достижения полной консистентности проект будущем может быть переписан, но пока это не приоритет из-за больших
> затрат ресурсов.

## Основная концепция

Главный принцип архитектуры:  
все файлы делятся на **локальные** и **глобальные**.

- **Локальные файлы** используют **принцип близости** — живут рядом с компонентом или страницей, внутри которых
  применяются.

- **Глобальные файлы** выносятся в `src/sharedSomething/` если они используются в нескольких местах проекта.

## **Структура файлов**

### **1. Глобальные модули**

В корне проекта находятся папки для переиспользуемых сущностей, доступных во всём приложении:

- `constants`: Глобальные константы.

- `middleware`: Файлы Middleware.

- `providers`: Глобальные провайдеры (например, React Context).

- `services`: Функции для работы с внешними API.

- `types`: Основные типы и интерфейсы TypeScript.

- `utils`: Вспомогательные функции (хелперы).

> ✍️ **Заметка**
> Папка `utils` также используется для вынесения логики и очистки JSX с целью повышения читаемости.
> **Правило:** если функции внутри компонента суммарно занимают >50 строк → их выносим в `utils`.

### **2. Локальные модули (принцип близости)**

Файлы, которые относятся только к одной странице или компоненту, располагаются максимально близко к месту использования.

**Правило:** Файлы `types`, `utils`, `constants` и `hooks` должны располагаться как можно ближе к компонентам, которые
их
используют.

- Если утилита нужна только в компоненте `DayHighlight` → она лежит в
  `app/(pages)/Home/components/shared/DayHighlight/utils`.

- Если хук нужен только в `Home`, кладём в `app/(pages)/Home/hooks`.

### 3. Константы

С константами действуют особые правила.

Мы выносим их в `src/constants/` **только тогда**, когда они несут **общеархитектурный смысл** — используются в разных
частях приложения или влияют на общий подход к архитектуре.

- Если в одном компоненте используется до 3 констант включительно → создавать отдельный файл не нужно

- Если константа нужна только в одном компоненте или странице → она остаётся локально, рядом или в коде.

- Если она определяет общие правила, маршруты, API или глобальные UI-настройки → выносится в `src/constants/`.

**Примеры глобальных констант:**

- `constants/routes.ts` — пути приложения

- `constants/api.ts` — базовые API-эндпоинты

- `constants/ui.ts` — глобальные UI-настройки

> ✅ **Как принять решение**

- **Локально:**  
  Константа описывает внутреннюю логику компонента (например, `DEFAULT_PAGE_SIZE` для конкретного списка).

>

	- **Глобально:**  
    Константа определяет **правила приложения в целом** (например, `ROUTES.HOME` или `API_TIMEOUT`).

>

	- **Сомнения:** 
    Если константа нужна и в родителе, и в дочерних компонентах → **оставляем рядом** с родительским модулем. Вынос в `constants/` оправдан только если этот код используется в разных, не связанных между собой частях приложения.

>

### **4. Файлы vs Папки**

**Это правило распространяется только на компоненты и типы (не касается utils, hooks, constants).**

- Если внутри сущности **1 файл** → создаём **только файл**.

- Если файлов **несколько** → создаём папку.

**Пример:**

```swift
app/(pages)/Home/components/mobile/
|
├── SearchCity/            // Это папка, т.к. есть дочерние файлы
│	├── SettingsDesktop.tsx          // Основной компонент
│	├── types.ts           // Типы только для SearchCity
│	└── compoents/         // Вложенные компоненты только для SearchCity
│		└── ...
│
├── MainWeatherDisplay.tsx // Cамодостаточный компонент
├── PanelBackground.tsx
└── WeeklyCard.tsx
```
